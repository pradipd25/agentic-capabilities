## Open

- [ ] Noticeable lag in response time when conversation is in a non-English language — likely Whisper taking longer to auto-detect language vs hardcoded EN
- [ ] LLM occasionally replies in the wrong language for a clearly English **text** prompt — observed during interruption testing: an English question ("Explain what Python is…") produced a French reply ("Python est un langage de programmation très…"). The `_BASE_INSTRUCTION` mirror rule is meant to force same-language replies, so this is the rule being applied unreliably by the model. Worth investigating: strengthen the instruction, or detect/pin the language of text input explicitly rather than relying on the model to infer it.

## Fixed

- [x] Avatar speaks only part of a sentence then stops, and often no reply comes — root cause: a **false barge-in** from speaker echo. The mic is always open and streams to the backend, where `VADProcessor` + `allow_interruptions=True` cancel the in-flight LLM/TTS on any detected speech. The only defenses were browser AEC (unreliable for Web-Audio-API playback, esp. on laptop speakers) and a single-chunk RMS gate, so residual echo crossed the threshold, fired the VAD, and killed the reply mid-sentence; the garbled echo produced no usable transcript, so no new reply came. Fixed with a **robust client-side gate**: capture chunk reduced 4096→1024 (256 ms→64 ms) and barge-in now requires `BARGE_IN_MIN_CHUNKS` (4) *consecutive* loud chunks above `BARGE_IN_RMS` (0.08, above the post-AEC echo floor). Echo is bursty so any quiet chunk resets the streak — transient echo never sustains; real (loud, continuous, close) speech does. Onset chunks are buffered during the streak and replayed on confirmation so the start of the utterance still reaches STT; the hold window is refreshed while speech continues. Confirms barge-in still interrupts and answers the new question (`test_bargein.py` → "Paris."). All thresholds are tunable constants in `App.tsx`. Headphones remain the most reliable option (zero echo).
- [x] Voice barge-in didn't work — had to wait for the avatar to finish before speaking. Caused by the (correct) echo fix above being fully half-duplex: it suppressed *all* mic audio during playback. Replaced the binary gate with an energy gate so the user can talk over the avatar: while TTS is playing, `handleChunk` forwards mic audio only when its RMS exceeds `BARGE_IN_RMS` (real speech is loud/close; post-AEC echo is quiet), with a `BARGE_IN_HOLD_MS` hangover so the whole utterance streams without choppy gaps. On the rising edge of barge-in the client also calls `useAudioPlayer.flush()` (stops all scheduled buffer sources, resets the clock) so the avatar goes silent immediately instead of draining buffered audio. Backend interruption path verified end-to-end (`test_bargein.py`): talking over a long count gets transcribed, interrupts, and answers the new question ("…Paris."). Thresholds are tunable constants in `App.tsx`.
- [x] Long responses go vocal briefly then cut off — root cause: a false VAD interruption from speaker echo. After the mic was opened during avatar speech (see below), residual echo (beyond what AEC removes) tripped the server VAD partway through a long reply, interrupting the TTS. Backend was confirmed innocent: a text-triggered long reply streams the full audio (1226 chars → ~84s of speech, no interruption — `test_long.py`). Fixed with a self-releasing mic gate: `handleChunk` now suppresses mic audio only while TTS is actually playing, driven by the audio clock (`isBotAudioPlaying` in `useAudioPlayer`, true while `nextTime > currentTime`). It releases the instant playback drains, so it can never stick (unlike the old `animation === 'talking'` flag) and the mic is open between turns for follow-ups / language switching.
- [x] No response when speaking — especially after switching language (e.g. to Hindi) mid-conversation. Root cause was NOT the backend: an end-to-end voice test that replays synthesized speech as mic input confirms Whisper transcribes and the LLM answers in both English and Hindi (`test_voice.py`). The fault was the frontend echo-prevention gate: `handleChunk` only sent mic audio while `animation !== 'talking'`. That hard-muted the mic whenever the avatar spoke, so (a) you could never barge in / switch language by voice, and (b) if `animation` ever stuck on `talking` (e.g. an interruption where `BotStoppedSpeaking` is delayed/dropped) the mic went dead permanently. Fixed by enabling the browser's acoustic echo cancellation (`echoCancellation`/`noiseSuppression`/`autoGainControl` in `useAudioCapture`) and replacing the sticky animation-based gate with the audio-clock gate described above. Echo no longer triggers the VAD, and any stray false trigger can no longer corrupt the conversation thanks to `ContextSanitizerProcessor`.
- [x] Avatar continues the previous (often wrong-language, e.g. Spanish) reply instead of answering an interrupting follow-up, then stops responding entirely — root cause: a context **ordering** bug. When a follow-up arrives mid-response, the new user turn is appended to the shared `LLMContext` *before* the interrupted reply is committed, leaving the context as `[…, user_followup, assistant_partial]`. A chat-completion request whose last message is an assistant message is treated as a *prefill* and the model continues that text instead of answering. Fixed with `ContextSanitizerProcessor` (placed just before the LLM) which drops any trailing assistant message(s) on each `LLMContextFrame` so generation always starts from the user's turn. Also changed `TextInputProcessor` to push an `InterruptionFrame` **downstream** (FIFO-ordered ahead of the new context) rather than an `InterruptionTaskFrame` upstream — the latter re-injects the interruption at the pipeline source where it can overtake and cancel the brand-new generation, wedging the pipeline. Verified: follow-up answered in English; turn 3 still responds; stable across repeated runs (`test_interruption.py`).
- [x] Avatar stops responding on voice follow-up — root cause: TTS audio playing through speakers was picked up by mic, triggering VAD and causing a false interruption that corrupted the LLM context. Fixed by suppressing mic audio to the WebSocket while `animation === 'talking'` (frontend gate in `handleChunk`).
- [x] Rate of speech too slow — root cause: Pipecat defaults to `gpt-4o-mini-tts` which silently ignores the `speed` parameter; fixed by switching to `model="tts-1"` which fully supports speed control. Default set to `tts_speed=1.25`; configurable via `TTS_SPEED` in `.env` (range 0.25–4.0)
- [x] Avatar responds only in English — two fixes applied: (1) LLM system prompt now instructs it to mirror the user's language; (2) WhisperSTT was hardcoded to `language=EN` causing Hindi speech to be transcribed as English text — fixed by setting `language=None` to enable Whisper's auto-detection. OpenAI TTS `tts-1` handles ~50 languages natively so spoken audio will also be in the correct language.
